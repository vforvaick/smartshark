# 2026-05-31 Evidence-First Product Doc Set

## Pattern
- For a new product concept with no codebase yet, keep `CONTEXT.md` as glossary-only and put product scope, schema, architecture, UX, permissions, and backlog in separate docs.

## Validated Outputs
- `docs/product-prd.md` for issue-ready PRD content.
- `docs/domain-model.md` for concept relationships.
- `docs/evidence-map-schema.md` for anti-hallucination schema and validator rules.
- `docs/architecture-overview.md` for MVP architecture and data flows.
- `docs/implementation-modules.md` for deep module boundaries and testing priorities.
- `docs/ux-workflows.md` for workspace and analysis UX.
- `docs/security-permissions-redaction.md` for local roles and AI data-sharing rules.

## Limits and Caveats
- Issue tracker publishing still requires a git repository or explicit target repository.
- ADRs should remain short; detailed product behavior belongs in supporting docs.
